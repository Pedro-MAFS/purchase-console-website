# 采购工作台官网与 GitHub 发布渠道 详细设计

## 背景与目标

### 需求来源

本文档依据用户与项目方在对话中已确认的**官网建设目标、品牌价值观、仓库与托管策略**整理为可实现、可验收的详细设计；未引用单独的需求 Markdown 文件。

### 业务背景

采购工作台为**闭源**产品，需面向潜在用户与现有用户提供**公开可访问**的品牌与获取入口。产品核心代码与官网/安装包分发在仓库层面**解耦**，以降低权限与泄露风险，并便于市场或外包仅参与官网仓协作。

### 目标

1. **宣传展示**：清晰呈现产品能力、亮点与适用场景，建立信任。
2. **提供下载**：为用户提供可验证、可追踪版本信息的安装包获取入口。
3. **价值观传达**：首屏与品牌叙事体现「勿以事小而不为」— 认真对待并响应用户的每一次反馈。

### 成功判据（摘要）

- 用户通过浏览器访问官网域名（或 GitHub Pages 默认域名）可完成「了解产品 → 找到下载入口 → 成功获取对应平台安装包」的主路径。
- 安装包来源为 GitHub Release 附件，官网不长期将大体积二进制纳入 Git 历史。
- 仓库内不出现产品核心业务源码；不出现密钥与内网配置。

---

## 范围与非目标

### 范围

- **独立 GitHub 仓库**：专门存放官网源码、站点构建配置、与官网相关的 CI；**同一仓库**通过 **GitHub Releases** 承载各版本安装包（附件），不将安装包作为主分支常规提交内容。
- **托管**：使用 **GitHub Pages** 发布构建后的静态站点。
- **分发**：使用 **GitHub Releases** 的 Assets 作为对外下载源；官网页面提供跳转或说明。
- **文案定稿**：首页价值观主标语与副标语采用本文「功能与交互」中的已定稿表述。

### 非目标

- 不在本文档范围内定义采购工作台**客户端/服务端**的功能需求与接口。
- 不实现需登录才可下载的复杂鉴权（若未来需要，另起专项设计）。
- 不包含应用内更新检查、差分更新等自动升级通道设计（可后续扩展）。

## 术语与约定

| 术语 | 含义 |
|------|------|
| 产品仓 | 存放闭源采购工作台核心工程的私有仓库（不在本设计展开）。 |
| 官网仓 | 与用户约定的**单独** GitHub 仓库：官网源码 + Release 发布包。 |
| Release | GitHub 发布版本，含版本号、说明与附件（Assets）。 |
| Pages | GitHub Pages，托管静态站点（通常指向某分支目录或 Actions 上传的 artifact）。 |

---

## 总体方案 / 架构要点

### 仓库边界

- **产品代码**：仅存于产品私有仓，**不**并入官网仓。
- **官网仓**：可设为**公开**仓库，以便匿名用户访问 Pages 与下载 Release 附件（闭源产品常见「公开下载、不公开源码」形态）。若组织策略要求仓库私有，需另行评估 Pages 与 Release 的可见性策略（可能需配合其他分发方式）；**本设计按「官网仓公开」落地**，以满足公开宣传与公开下载。

### 托管与分发

| 能力 | 实现 |
|------|------|
| 站点托管 | GitHub Pages，发布内容为**构建后的静态文件**（非直接把未构建的复杂源码暴露为站点根）。 |
| 安装包 | 仅通过 **Releases → Assets** 提供；由维护者在发版时上传，或由 CI 在推送版本标签时构建并上传附件。 |
| Git 内容 | **禁止**将大体积安装包作为常规提交进入默认分支历史；避免仓库膨胀与克隆成本上升。 |

### 技术栈（单一实现路径）

- 官网采用 **Vite 5** 与 **TypeScript** 搭建静态导向站点（可为多页或单页应用，以首屏营销 + 下载页为主）。
- 构建命令统一为 `npm run build`，输出目录为 **`dist/`**。
- 使用 **GitHub Actions** 工作流：在推送到约定分支（如 `main`）且变更满足触发条件时，执行安装依赖、构建，将 `dist/` 部署到 GitHub Pages（例如使用社区维护的 `peaceiris/actions-gh-pages` 或官方 `actions/upload-pages-artifact` + `actions/deploy-pages`，具体以仓库启用 Pages 时选择的来源为准，**仅保留一种**部署链并在仓库内固化）。

### 域名

- **默认**：使用 `https://<owner>.github.io/<repo>/`（或组织 Pages 对应规则）。
- **自定义域名**：若后续绑定自有域名，在 Pages 设置中配置 CNAME，并在 DNS 完成指向；不在本文档强制阶段内。

## 功能与交互（含状态与边界）

### 信息架构（最小集）

1. **首页**  
   - 首屏展示价值观与产品一句话定位。  
   - 主行动：进入「功能/产品」或「下载」（按钮或导航）。  
2. **功能/产品**（可与首页二级屏合并为长页，视实现而定）  
   - 分块介绍核心能力，配图或占位图；**不**暴露源码或内部架构细节。  
3. **下载**  
   - 按平台（如 Windows / macOS，以实际发版为准）列出推荐版本、Release 说明链接、各 Asset 下载按钮或外链。  
   - 可选：展示 SHA256 校验和（若 Release 说明或单独文件中提供），增强可信度。

### 首页价值观（已定稿文案）

- **主标语**：勿以事小而不为  
- **副标语**：再小的反馈也值得认真对待。

英文站若后续需要，可另行增加 i18n；**首版仅中文**。

### 边界与异常

- Release 尚未发布任何版本时：下载页展示「即将发布」或引导至联系方式 / Issue，**不**出现死链按钮。  
- 某平台暂无安装包：对应平台区域显示「敬请期待」或隐藏该平台块，**不**链接到空 Asset。  
- 外部用户无法访问私有 Release：本设计假定官网仓为**公开**且 Release 附件可匿名下载；若策略变更，下载页需同步改为带鉴权链接或外链存储（属范围外变更，需另案）。

## 数据模型与契约（字段、校验、兼容）

本站为静态营销站，无自有业务数据库。与「版本」相关的展示建议约定如下**展示契约**（可由构建时注入或维护小 JSON，二选一，实现阶段选一种即可）：

| 字段 | 说明 |
|------|------|
| `latestVersion` | 当前推荐版本号，与 GitHub Release tag 一致（如 `v1.0.0`）。 |
| `releaseNotesUrl` | 指向对应 Release 页面的 URL。 |
| `artifacts` | 数组：项含 `platform`、`label`、`downloadUrl`（指向具体 Asset 的直链）。 |

**校验**：`downloadUrl` 必须为 `https://github.com/` 同源下的 Release 资产路径或官方跳转，**禁止**在前端硬编码非 GitHub 的临时网盘链接作为唯一渠道（除非产品决策变更）。

## 接口与模块职责

| 模块 | 职责 |
|------|------|
| 静态页面与组件 | 展示文案、版式、导航与下载区。 |
| 构建脚本 | 产出 `dist/`，可注入环境变量（如站点 base path）以适配 `/repo/` 子路径部署。 |
| CI（GitHub Actions） | 依赖安装、构建、部署 Pages；可选第二条 workflow：在 `v*` tag 推送时触发（若安装包在同一仓内由 CI 构建）。 |
| Release（人工或 CI） | 维护版本说明与二进制附件；为下载页提供稳定 URL。 |

## 错误处理与可观测性

- 构建失败：Actions 失败时通知维护者（邮件或仓库通知），**不**部署半成品到 Pages。  
- 下载链失效：定期人工巡检下载页链接与最新 Release 是否一致；可选在 CI 中加简单链接探测（非强制）。  
- 不在官网仓存放应用日志或用户数据。

---

## 验收与测试要点

1. 访问 Pages 地址，首页可见主标语与副标语，文案与本文「首页价值观」一字不差（标点除外可统一全角）。  
2. 从导航可进入下载页，且链接指向**当前公开 Release** 中存在的 Asset（或明确「即将发布」状态）。  
3. 仓库默认分支中无大于 GitHub 建议阈值的大文件提交（安装包仅存在于 Releases）。  
4. 全站无产品私有源码、无 API Key、无内网 URL。  
5. 在主流浏览器（Chromium、Firefox、Safari 各选一最新稳定版）完成首页与下载页冒烟测试。

## 待确认（仅在有未决问题时保留）

以下事项未在对话中明确，实现前需产品或维护者拍板：

1. **官网仓命名与 GitHub Owner**：最终仓库 URL 与组织/个人账号。  
2. **发版平台矩阵**：首期是否仅 Windows、是否包含 macOS/Linux。  
3. **安装包是否由 CI 在同一官网仓内构建**：若安装包仅在本地构建后手动上传 Release，则无需在官网仓内配置构建产物的复杂流水线。  
4. **自定义域名与备案**：若面向中国大陆用户，是否使用国内备案域名与 CDN，与纯 GitHub Pages 策略可能不同，需单独决策。
